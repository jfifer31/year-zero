# Contributing

Year Zero Education welcomes serious contribution, critique and implementation feedback.

The project is open, but not casual. It is designed to produce a traceable Version 0.1 curriculum hypothesis, not to collect unstructured opinions.

---

## 1. Ways to contribute

Useful contributions include:

- challenging a claim;
- correcting a source;
- adding better evidence;
- identifying missing perspectives;
- improving public explanation;
- improving accessibility;
- testing a learning activity and reporting what happened;
- identifying safeguarding, privacy or implementation risks;
- improving templates or repository usability;
- proposing curriculum improvements once Stage 10 is active.

---

## 2. Contribution types

When contributing, identify the type of contribution.

| Type | Example |
|---|---|
| Research challenge | “This claim is not supported by the cited source.” |
| Source suggestion | “This review should be considered in Stage 1.” |
| Assumption challenge | “This assumption may not hold for low-resource contexts.” |
| Implementation feedback | “We tried this with a home-educated learner and this part was unclear.” |
| Ethical concern | “This proposed evidence collection could expose private learner data.” |
| Documentation improvement | “This file repeats another file and should be merged.” |
| Curriculum suggestion | “This project format may better develop the stated capability.” |

---

## 3. Evidence and claim expectations

A contribution does not need to be academic to be useful, but it must be clear about what kind of support it provides.

Evidence may include:

- peer-reviewed research;
- official reports;
- high-quality books or reviews;
- practitioner evidence;
- implementation observation;
- learner or parent feedback;
- reasoned critique;
- technical analysis;
- public communication feedback.

Do not present anecdote as proof. Practical experience is valuable, but it should be described as experience rather than universal evidence.

---

## 4. Claim discipline

If your contribution affects a claim, identify the claim type and whether your change affects evidence, confidence, uncertainty, downstream use or revision triggers.

Use the full claim taxonomy and handling rules in `docs/RESEARCH_OPERATING_SYSTEM.md`; this file does not duplicate them.

---

## 5. How to challenge a claim

Use the research challenge issue template.

A strong challenge should include:

1. the file and section affected;
2. the specific claim being challenged;
3. why it may be wrong, overstated or incomplete;
4. better evidence or reasoning;
5. the proposed correction;
6. whether the issue affects downstream curriculum design.

---

## 6. How to provide implementation feedback

Use the implementation feedback issue template.

Useful implementation feedback should include what was tried, broad age range, broad setting, what worked, what confused users and what should change.

Do not include private learner information, learner work, images, videos or AI logs. Follow the full safety rules in `docs/ETHICS_AND_PUBLIC_TESTING.md`.

---

## 7. Privacy and child-safety rules

Do not submit private learner data or anything that could identify, expose, rank, shame or promote a learner. Use broad, anonymised summaries only.

Detailed safeguarding, privacy, consent/assent, learner-data, AI-log, public-testing and promotional-use rules are in `docs/ETHICS_AND_PUBLIC_TESTING.md`.

---

## 8. Conduct expectations

Contributors should:

- criticise ideas, not people;
- be precise;
- be willing to update their view;
- identify uncertainty;
- avoid ideological capture;
- avoid performative certainty;
- respect the seriousness of work involving young people.

The project welcomes disagreement. It does not welcome harassment, personal attacks, bad-faith argument or unsafe publication of private information.

---

## 9. Pull request expectations

A pull request should explain:

- what changed;
- why it changed;
- which files are affected;
- what evidence or reasoning supports the change;
- what risks or uncertainties remain;
- whether the change affects research, curriculum, public explanation or repository structure.

Large rewrites should first be discussed through an issue unless they are part of an agreed project phase.

---

## 10. Curriculum contributions

Before Stage 10, contributors should not submit curriculum content as if the curriculum architecture already exists.

Before Stage 10, useful curriculum-adjacent contributions include:

- evidence about learning;
- examples of capability development;
- critiques of assumptions;
- implementation constraints;
- comparisons with existing models.

After Stage 10, curriculum contributions should map proposed learning experiences back to:

- derived capabilities;
- knowledge/experience/practice mappings;
- evidence-of-learning architecture;
- limitations and assumptions.

---

## 11. Repository structure contributions

The repository follows a low-sprawl principle.

Do not add a new file if the content belongs naturally inside an existing canonical file.

A new file is justified only if:

- it serves a distinct user journey;
- it will be maintained independently;
- merging it would make another file too difficult to use;
- agents or contributors need it as a clear operating object.

---

## 12. Review principle

The best contributions make the project more:

- truthful;
- usable;
- safe;
- traceable;
- understandable;
- testable;
- open to improvement.

They do not simply make the project sound more confident.
